# Web I18n Reference

## Scope
- INCLUDE: French locale namespaces, user-facing copy, React i18next usage, locale compilation, generated-output guardrails, and validation of translation changes.
- INCLUDE: Decisions about namespace placement, key naming, interpolation, and copy updates for routes, forms, hooks, and feature components.
- EXCLUDE: Backend mail copy, API validation implementation, generated locale bundles, and design-token naming.
- EXCLUDE: Adding a new language unless the task explicitly asks for it.

## Workflow
- Step 1: Locate the nearest French locale namespace for the route, feature component, hook, or shared app surface being changed.
- Step 2: Add or update keys in the source locale file, keeping names close to the user-facing concept.
- Step 3: Read copy through `useTranslation` in components or hooks rather than passing raw strings across many layers.
- Step 4: Run the locale compiler so source namespaces are merged into the generated i18n bundle.
- Step 5: Validate that changed screens, form errors, toasts, and empty states resolve keys instead of showing fallback key text.

## Required Rules
- Write user-facing copy in French source locale namespaces. Don't hard-code French strings in JSX, hooks, or mutation options.
- Keep namespaces local to the feature or route that owns the copy. Shared app copy belongs in a shared app namespace.
- Treat compiled i18n output as generated. Never edit it by hand.
- Keep interpolation explicit and named. Don't build translated sentences by concatenating strings.
- Use React i18next hooks in the component or hook that owns the displayed message.
- Run the i18n build or watcher after changing locale source files.
- Keep validation and mutation messages aligned with the form or action that shows them.

## Key Considerations
- Namespace placement is part of ownership. Route titles, tab labels, and page-specific empty states should stay near the route surface.
- Hook-level messages such as toast text can live with the hook namespace when the hook owns the side effect.
- Form labels, descriptions, and validation messages should sit near the form component or form hook that uses them.
- The compiler merges French locale sources into one generated resource for i18next. Missing or duplicate keys are easier to catch before that merge.
- Typed i18n resources depend on the generated French bundle, so compile before checking types after copy changes.
- Prefer short, natural French copy that matches the product voice. Don't translate technical field names literally when a clearer label exists.

## Examples
**Do**
```json
{
	"profile": {
		"form": {
			"name": {
				"label": "Nom"
			}
		}
	}
}
```

```tsx
const { t } = useTranslation();
return <field.TextField label={t("profile.form.name.label")} />;
```

**Don't**
```tsx
return <field.TextField label="Nom" />;
```

## Anti-Patterns
- Editing compiled i18n JSON instead of source namespaces.
- Adding copy to a far-away namespace because it already has a similar key.
- Concatenating translated fragments around variables.
- Leaving raw translation keys visible in UI after changing source JSON.
- Mixing English fallback strings into French-only user-facing flows.
